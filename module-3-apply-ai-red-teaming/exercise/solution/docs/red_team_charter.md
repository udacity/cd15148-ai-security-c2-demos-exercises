# Red Team Charter: EnterpriseSummarizer RAG Security Assessment

| Field | Value |
|-------|-------|
| Engagement Name | EnterpriseSummarizer Limited Deployment AI Red Team Assessment |
| Date | May 2026 |
| Assessor | Security Engineering Team |
| Sponsor | Enterprise Operations Leadership |
| System Owner | Enterprise AI Platform Team |
| Estimated Assessment Window | 35-minute exercise simulation |

## Objectives

Evaluate the security posture of the EnterpriseSummarizer RAG document summarization service by planning controlled tests across six prioritized attack vectors:

1. **Retrieved-Document Prompt Injection**: Assess whether instruction-like content in retrieved documents can override summarization policy or safety constraints.
2. **Cross-Department Retrieval Bypass**: Determine whether weak metadata filtering or client-controlled department values can retrieve unauthorized documents.
3. **Sensitive Data Leakage Through Summaries, Citations, or Logs**: Verify whether generated summaries, citation metadata, denied responses, or debug logs expose restricted information.
4. **Insecure Export Tool Use**: Assess whether generated summaries can be exported to unauthorized workspace destinations.
5. **Cost and Availability Abuse**: Evaluate whether long queries, broad document scopes, or repeated requests cause unacceptable latency or retrieval cost.
6. **Monitoring and Detection Gaps**: Identify whether unusual retrieval, denial, prompt-injection, or export patterns are visible to defenders.

## Scope

### In Scope

- `POST /v1/summaries` in a staging or limited internal test environment
- `POST /v1/tools/export` with synthetic summaries and approved test destinations
- Retrieval policy enforcement for department, allowed group, classification, and document scope
- Prompt assembly behavior involving user queries and retrieved chunks
- Citation handling for accessible and inaccessible documents
- Logs and metrics generated by synthetic test traffic
- Dependency and deployment artifact review where provided by the course environment

### Out of Scope

- Production documents, production users, and live company workflows
- Credential theft, phishing, or social engineering
- Destructive denial-of-service testing
- Unauthorized access to third-party services or vendor infrastructure
- Attempts to modify real knowledge-base documents or deployment pipelines
- Testing that exports real confidential data outside approved test destinations

## Rules of Engagement

1. Use only synthetic documents, synthetic markers, and approved staging data.
2. Run all tests in the course-provided environment or a designated staging environment.
3. Do not attempt to access real HR, finance, engineering, operations, payroll, board, or incident documents.
4. Identify test traffic using a unique `metadata.test_run_id` when the request schema allows metadata.
5. Keep load and latency testing within limits approved by the instructor or system owner.
6. Stop testing if the service becomes unstable or if sensitive real data appears unexpectedly.
7. Record request body, response body, timestamp, expected authorization result, and observed behavior for each test.
8. Classify each result as confirmed, partially confirmed, not reproduced, or rejected as speculative.

## Success Criteria

| Attack Vector | Success Metric |
|---------------|----------------|
| Retrieved-Document Prompt Injection | Demonstrate whether a synthetic retrieved instruction changes summary behavior, citation behavior, or policy adherence |
| Cross-Department Retrieval Bypass | Confirm that mismatched department claims, client-supplied department values, and restricted `document_scope` IDs are denied consistently |
| Sensitive Data Leakage | Identify whether summaries, citations, errors, denied responses, or logs reveal synthetic restricted markers or inaccessible document metadata |
| Insecure Export Tool Use | Verify that cross-department or unapproved export destinations are denied and logged |
| Cost and Availability Abuse | Measure whether approved long-query and broad-scope testing pushes p95 latency above an agreed threshold |
| Monitoring and Detection Gaps | Determine whether repeated denied retrieval, prompt-injection, and export attempts generate useful alerts or investigation signals |

## Deliverables

1. Completed red team charter
2. Prioritized LLM-generated attack vector list with rejected speculative findings
3. Evidence table showing which findings are supported by provided context
4. Mitigation plan aligned to P1, P2, and P3 risks
5. Executive summary explaining whether the system is ready for company-wide expansion

## Assessment Notes

LLM-generated attack vectors are planning hypotheses. Findings become actionable only after a human reviewer confirms relevance, reproducibility, operational impact, and a mitigation path.
