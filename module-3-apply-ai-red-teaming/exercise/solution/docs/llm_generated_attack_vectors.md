# LLM-Generated Attack Vector Candidates

The table below is a sample LLM-style output generated from the structured prompt and RAG system context. These are proposed vulnerabilities for red team planning, not confirmed findings.

| Priority | Attack Vector | Targeted Component | Supporting Evidence | Example Test Idea | Expected Impact If Confirmed | Confidence | Mitigation Ideas | False-Positive or Validation Notes |
|----------|---------------|--------------------|---------------------|-------------------|------------------------------|------------|------------------|------------------------------------|
| P1 | Retrieved-document prompt injection overrides summarization policy | RAG prompt assembly and hosted LLM endpoint | Model card says retrieved chunks may contain instructions, templates, comments, or copied email text; prompt injection testing has been limited | Add a synthetic authorized document chunk containing instruction-like text and verify whether the summary follows retrieved instructions instead of system policy | Restricted content may be summarized, citations may be manipulated, or safety instructions may be bypassed | High | Separate retrieved content from instructions, add prompt-injection filters, use strict system prompts, evaluate with adversarial documents | Confirm with synthetic documents only; do not modify real knowledge-base content |
| P1 | Cross-department retrieval through weak metadata filtering | Retrieval policy filter and vector database | Retrieval relies on `department`, `allowed_groups`, and `classification`; metadata is maintained by document owners and may be incomplete | Submit requests with mixed `department` and `document_scope` values and verify that inaccessible finance or HR documents are not retrieved | Unauthorized access to confidential HR, finance, or executive planning content | High | Enforce authorization server-side from identity claims, reject client-supplied department values, add deny-by-default metadata checks | A 403 response in one sample is positive evidence, but coverage across metadata combinations is unknown |
| P1 | Sensitive data leakage through summaries or citations | Summarization API response and citation rendering | API returns citation `doc_id`, title, and classification; policy says citations should not reveal inaccessible documents | Request summaries with inaccessible `document_scope` IDs and inspect whether citations, titles, or partial summaries reveal restricted document existence | Users may infer restricted document names, classifications, or sensitive facts | Medium | Suppress inaccessible citation metadata, normalize denied responses, add retrieval result post-filtering | Must distinguish allowed citations from leaked inaccessible citations |
| P2 | Export tool can write sensitive summaries to unauthorized destinations | Workspace export tool | Export authorization is application-enforced; no separate export rate limit; export destination should match department | Attempt approved test exports to same-department and cross-department destinations using synthetic summaries | Summaries containing confidential content may be moved outside intended department controls | Medium | Bind export destinations to identity claims, require destination allowlists, log and alert cross-department attempts | Test only synthetic summaries and approved staging destinations |
| P2 | Debug prompt logging captures sensitive retrieved text | Logging and monitoring pipeline | Infrastructure notes say prompt text is sampled into debug logs at 2 percent during limited deployment | Submit synthetic sensitive markers in authorized documents or prompts and inspect whether full prompt or retrieved chunks appear in logs | Sensitive document text may be exposed to log readers or retained longer than intended | High | Disable prompt sampling for sensitive workflows, redact retrieved chunks, restrict log access, set retention limits | Use synthetic markers to avoid exposing real sensitive content |
| P2 | Excessive retrieval or long-query abuse increases cost and latency | API gateway, retrieval layer, LLM endpoint | Query max length is 6000 characters; top-k retrieval defaults to 8; rate limit is 120 requests per user per hour | Send approved bursts of long queries across maximum `document_scope` values and measure p95 latency and error rate | Degraded availability and high LLM or vector database cost during broader rollout | Medium | Add cost-aware limits, cap retrieved tokens, tune top-k, add per-user and per-department quotas | Keep load testing within approved staging limits |
| P2 | Client-supplied department value causes authorization confusion | API validation and retrieval policy | Request schema requires `department`; architecture says department claims are forwarded from gateway | Submit requests where token claim department differs from request body department and verify server-side enforcement | Users may retrieve documents outside their true department if body values are trusted | High | Ignore client-supplied department for authorization, derive department from signed claims, log mismatches | Requires test tokens or mocked claims to validate properly |
| P3 | Monitoring misses unusual retrieval and export patterns | Detection and response controls | Alerts exist for high error rate and latency, but not unusual retrieval patterns | Run benign staged requests with repeated denied scopes, broad document scopes, and export attempts; check alert coverage | Slow data discovery attempts may go undetected | Medium | Add alerts for denied retrieval spikes, cross-department scope attempts, and unusual export destinations | Lack of an alert is a control gap, not direct evidence of exploitation |
| P3 | Container or dependency drift creates RAG supply chain risk | CI/CD and container registry | Container image is built weekly from pinned dependencies; hosted model is accessed through proxy | Review dependency scan, image provenance, and proxy configuration for unpinned or vulnerable components | Compromised dependency or proxy misconfiguration could expose prompts or alter summaries | Medium | Enforce SBOMs, image signing, dependency scanning, and model proxy configuration reviews | Requires artifact access not included in API-only testing |

## Rejected or Deferred Candidates

| Candidate | Reason to Reject or Defer |
|-----------|---------------------------|
| Direct employee credential theft | The exercise materials do not describe login flows or credential handling, and credential attacks are out of scope. |
| Production data exfiltration | Testing must use synthetic or approved staging data only. |
| Full model extraction | The service is a hosted generative model behind a RAG application; the context does not support model extraction as a near-term assessment objective. |
| Training data poisoning | No live model training or document ingestion path is described. Treat document prompt injection as the relevant in-scope content manipulation risk. |
